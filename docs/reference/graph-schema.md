# Cortex graph schema

Source of truth for the graph DB shape. Every memory file's frontmatter conforms to these conventions; every typed query in `server/graph/` reads from these tables; migrations only add — they never reshape — once the design lands.

Two principles drive the design:

1. **Outcomes and authorship are first-class.** Every memory carries who/what produced it and how it ended. Promotion gates filter on these. Hallucinated content can't easily forge a real `outcome_text` because that field is copied verbatim from a transcript by deterministic code, never written by an LLM.
2. **Verbatim anchoring beats LLM paraphrase.** Where a JSONL transcript provides the raw text (tool args, error messages, exit codes), the graph stores it verbatim. The LLM's contribution is *categorization* and *grouping* — labels and topic names — not generated content.

---

## Node tables

### `Entry` (generic — kind-discriminated)

Every memory, decision, reference, workflow, error note, topic, concept, file, tool, and session lives here. The `kind` column distinguishes shape; specialized tables (below) hold structured fields for kinds that have them.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | Stable identifier. Conventions: `pattern:<projDir>/<sid>/<failId>` is deprecated → file-derived `<encProj>/pattern-<fp>`. Concept ids: `concept:<slug>`. Session ids: `session:<encProj>/<sid>`. File ids: `file:<absPath>`. Tool ids: `tool:<name>`. Free-form ids for memories. |
| `name` | STRING | Human-readable title |
| `kind` | STRING | One of: `memory`, `decision`, `reference`, `workflow`, `error`, `topic`, `note`, `concept`, `pattern`, `tool`, `file`, `session`, `agent` |
| `description` | STRING | One-line summary |
| `content` | STRING | Body markdown, capped at 8 KB |
| `source` | STRING | Path to the .md file on disk (or `session:<id>` for live extracts) |
| `scope` | STRING | `user`, `project:<encProj>`, `pattern:auto`, `tool`, `vault:<name>`, `shared:<name>`, `session:<id>` |
| `updatedAt` | INT64 | mtime ms |
| `syncedAt` | INT64 | Last graph sync ms |
| **`authorship`** | STRING | `human`, `agent`, `mixed`, `auto-extracted`. Default `auto-extracted` for synced files lacking the field. |
| **`outcome`** | STRING | `success`, `failure`, `unknown`. NULL/empty when not applicable. |
| **`outcome_text`** | STRING | Verbatim outcome string from the transcript. Error message text, exit code line, success result. Never LLM-paraphrased. |
| **`agent_id`** | STRING | Set by the orchestrator (Phase 9) when an agent authored the memory. Empty for human-authored. |
| **`session_id`** | STRING | Originating Claude Code session ID. Empty when derived from non-session sources (vault imports, manual writes). |
| **`pinned`** | BOOL | Frontmatter `pinned: true` → load-bearing memory. +0.3 recall-score boost; overrides observation stale-trend. Migration `0005`. |
| **`machine`** | STRING | Lineage — which machine authored / first-synced the memory. From frontmatter `machine:` (preserved verbatim across a private-mind adopt) or the local machine-id at sync time. Migration `0006`. |

Bolded columns are additive, all NULL/empty-tolerant (migrations `0002`, `0005`, `0006`).

### `Session` (specialization — Claude Code session lifecycle)

Every Claude Code session gets a Session entry in `Entry` (with `kind = 'session'`) plus a row here for structured lifecycle fields. The Entry row holds the human-readable `name` and `description`; this table holds quantitative state.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | Same value as the corresponding `Entry.id` (`session:<encProj>/<sid>`) |
| `started_at` | INT64 | First-seen timestamp (SessionStart) |
| `ended_at` | INT64 | SessionEnd timestamp; 0 while session is active |
| `turns_count` | INT64 | Recorded turns in the JSONL |
| `files_touched_count` | INT64 | Distinct files referenced in tool args |
| `tools_used_count` | INT64 | Distinct tools used |
| `final_state` | STRING | `active`, `complete`, `abandoned` |
| `auto_named` | BOOL | True when the session name was generated by `--auto`; false when user-provided. Extractor only overwrites the name when this is true. |

**Session naming model.** Claude Code's session UUID is the **stable identifier**, never reusable, not human-readable. The Session entry's `name` field is a separate **human label** that can be set, changed, and reset without affecting any memory's `session_id` (which holds the UUID). This decouples identity from label:

- A memory's `session_id` is the UUID — never changes when the session is renamed
- Queries that resolve "which memories are in the X session" work by traversing `OCCURRED_IN` to the Session entry, matching on `name`
- Renaming a session updates exactly one column on one row; no cascade

**Naming model (current).** Session topics use Claude Code's **native `custom-title` event** in the session JSONL — the same mechanism as `claude -n "<name>"`, which Claude Code propagates across `--resume`/`-c` automatically. `/cortex-rename <name>` appends a `custom-title` event via `bin/ckn-name-session.ts`; `ckn-extract` reads the latest one at SessionEnd to set the Session entry's `name`. The old turn-5 `prompt_state` state machine (`pending`/`prompted`/`named`/`skipped`) was **removed** (commit `20cfb69`) — there is no naming prompt and `auto_named` is retained only for backward compatibility on old rows.

### `Observation` (specialization — auto-consolidated beliefs)

Derived by `bin/ckn-derive.ts` (`POST /api/derive`) from clusters of semantically-similar memories. The `Entry` row has `kind = 'observation'`; this table holds the consolidation signals.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | `observation:<scope>/<slug>-<count>` |
| `trend` | STRING | `stable` / `strengthening` / `weakening` / `stale` — recency of supporting evidence |
| `evidence_count` | INT64 | Number of source memories in the cluster |
| `first_observed` | INT64 | Earliest supporting memory mtime |
| `last_observed` | INT64 | Most recent supporting memory mtime |
| `observer` | STRING | `self` by default; reserved for Honcho-style observer/observed pairing in the shared-mind future |
| `pinned` | BOOL | User-confirmed; sorts to top of capability-sheet surfacing, overrides stale |

Migration `0004`. Pattern-kind entries are excluded from clustering (they have their own recall path).

### `Pattern` (specialization — fail→success structured fields)

Pattern entries also live in `Entry` (one row per file). This table holds the structured columns that would be NULL for every non-pattern row in `Entry`.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | Same value as the corresponding `Entry.id` |
| `tool` | STRING | Tool name (`Bash`, `Edit`, `mcp__databricks__execute_sql`) |
| `fail_args` | STRING | Verbatim args from the failing tool_use |
| `success_args` | STRING | Verbatim args from the succeeding tool_use |
| `error_text` | STRING | Verbatim error from the failing tool_result |
| `fingerprint` | STRING | FNV-1a hash for semantic dedup |

Patterns are joined via `id` to their `Entry` row; recall queries hit `Pattern` directly when filtering by tool.

### `Symbol` (specialization — code-graph / AST tier)

Code symbols (functions, classes, methods, modules, interfaces, …) extracted from
source by the companion `codegraph` package and ingested via
`POST /api/graph/symbols/upsert`. A **separate node type** from `Entry` — code
symbols are derived/regenerable data with their own lifecycle, kept apart from the
memory graph. Migration `0007`.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | `<repo>:<file>#<name>` (e.g. `cortex:server/graph/db.ts#getConnection`) |
| `name` | STRING | Symbol name; module nodes use the file path as the name (id keeps `#<module>`) |
| `symbolKind` | STRING | `function` / `class` / `method` / `module` / `interface` / `variable` / `type` |
| `repo` | STRING | Source repo — the unit of forget/federation |
| `file`, `lang`, `line` | STRING/STRING/INT64 | Source location |
| `signature` | STRING | Optional signature text |
| `base`, `stickiness`, `centrality`, `lastSeen` | DOUBLE/DOUBLE/INT64/INT64 | Lifecycle: `centrality` is in-degree (who depends on it); preserved across re-extracts |
| `pinned`, `groundTruthValid` | BOOLEAN | `groundTruthValid:false` = provably stale (vanished from a re-extract snapshot) but not deleted |
| `syncedAt` | INT64 | Last upsert ms |

Re-ingest with `reExtractedRepos:[repo]` flips symbols absent from the new snapshot
to `groundTruthValid:false` rather than deleting them; `forgetRepoSymbols(repo)`
(via `POST /api/graph/symbols/forget`) deletes the whole repo subgraph (its symbols
and their edges) and tombstones its federated `codegraph/<repo>/graph.json`.

### `Session` presence columns — session bus (migration `0008`)

The bus is the **sole writer** of these columns. They represent live session presence and are overwritten on each heartbeat/register call; the Session specialization table still carries the structured lifecycle fields from migration `0003`.

| Column | Type | Notes |
|---|---|---|
| `friendly_name` | STRING | Human-readable name for this session (e.g. a topic slug or `/cortex-rename` value) |
| `cwd` | STRING | Working directory at registration time |
| `machine` | STRING | Machine ID where the session is running (same lineage stamp as `Entry.machine`) |
| `title` | STRING | Window/terminal title; updated on each heartbeat |
| `last_seen` | INT64 | Epoch ms of the last heartbeat; 0 until first register |
| `status` | STRING | Persisted: `live` (active session) / `signed_off` (graceful exit). Derived at read time by `presenceStatus()`: `idle` (last_seen > 5 min) / `stale` (last_seen > 60 min) — NOT stored. |
| `supersedes` | STRING | ID of an earlier session this one continues (resume chain) |

### `BusMessage` (session-to-session communication)

Append-only message store for session-to-session communication. Readers poll via `GET /api/bus/inbox/:session`; the bus marks delivery via `delivered_to` and acknowledgement via `acked_by`. `to_addr` is named to avoid `to` as a column name. `delivered_to` and `acked_by` are comma-joined session IDs (not list columns — this repo uses no list columns). Migration `0008`.

| Column | Type | Notes |
|---|---|---|
| `id` | STRING (PK) | `m_<nanoid(10)>` (e.g. `m_V1StGXR8Z5`) |
| `from_session` | STRING | Sender session ID |
| `from_name` | STRING | Sender `friendly_name` at send time (denormalised for cheap display) |
| `to_addr` | STRING | Recipient address: a session ID, a friendly name, or `*` (broadcast) |
| `kind` | STRING | `msg` \| `ack` \| `reply` \| `done`. Broadcast is expressed by `to_addr = '*'`, not by a separate kind. |
| `ref` | STRING | Optional reference — e.g. a ticket ID or Cortex memory ID |
| `body` | STRING | Message body (plain text or JSON string, ≤ 4 KB) |
| `created_at` | INT64 | Epoch ms when the message was created |
| `delivered_to` | STRING | Comma-joined session IDs that have received this message |
| `acked_by` | STRING | Comma-joined session IDs that have acknowledged this message |
| `status` | STRING | `open` \| `acked` \| `done` |

## Edge tables (typed)

Every meaningful relationship is a separate typed edge table. Replaces the old `LINKS_TO` table whose `label` field carried the type as a string.

| Edge | FROM → TO | Properties | Semantics |
|---|---|---|---|
| `MENTIONS_FILE` | Entry → Entry (kind=file) | `weight FLOAT` | Memory mentions a specific file path |
| `MENTIONS_TOOL` | Entry → Entry (kind=tool) | `weight FLOAT` | Memory mentions a specific tool |
| `RESOLVES` | Entry → Entry (kind=error) | `confidence FLOAT` | Pattern/decision resolves an error |
| `CONTRADICTS` | Entry → Entry | `notedAt INT64` | Memory contradicts another memory |
| `OCCURRED_IN` | Entry → Entry (kind=session) | — | Memory was extracted from a session |
| `AUTHORED_BY` | Entry → Entry (kind=agent) | — | Memory was authored by an agent identity |
| `EVOLVED_INTO` | Entry → Entry | `notedAt INT64` | Memory was superseded by a newer memory |
| `DERIVED_FROM` | Entry (kind=observation) → Entry | — | Observation was synthesized from this source memory (one edge per cluster member). Migration `0004`. |
| `LINKS_TO` | Entry → Entry | `label STRING` | **Deprecated.** Kept for `inferNameMentionEdges` output until that pass is removed. New code should not write here. |
| `SURFACED_IN` | Entry (memory) → Entry (kind=session) | `weight FLOAT` (surface count), `notedAt INT64` (last-surfaced-at, ms) | s1 surfacings log: this memory was recalled/surfaced into that session. One edge per (memory,session) — repeats upsert `weight = weight+1`, `notedAt = now` (no per-event rows). Written by `recordSurfacings` from the recall routes; read by s3 (acted-on correlation: compose with `MENTIONS_FILE` + `OCCURRED_IN`) and s4 (decay). **OBSERVATIONAL** — see below. |

**Observational edges (system-observed, not file-declared).** `SURFACED_IN` is recorded at runtime by the system, not declared in any memory's frontmatter, so it has no source to re-derive it. A memory's re-upsert (`upsertEntry`) wipes the entry's *declared* outbound edges but must EXCLUDE this class, or the observed history resets on every edit (and `/cortex-snapshot` rewrites memory files constantly). The excluded set is `OBSERVATIONAL_RELS` in `server/graph/sync.ts`; new runtime-observed rels join it there.

**Thread-strip resilience (the graph holds authority a degraded file can't override).** A `thread` `.md` is an exchange surface any external writer can mangle — e.g. a tool that tidies frontmatter to the harness memory shape (`name` / `description` / `metadata.type` / `body`) silently drops a thread's `status` / `next_step` / `links` / `id`. No Cortex serializer produces those bytes (`serializeMemory` provably preserves the metadata block), so the defense lives at the sync chokepoint, not in a serializer. On each changed file `syncMemories` looks up any prior `thread` entry by `source` (the file path — the one identity a strip cannot remove, so it covers both the id-intact and id-dropped strip shapes). If that prior thread **has state** and the incoming re-read either **lost thread-kind** (Branch 1) or **hollowed** to `status=open` + empty `next_step` (Branch 2), the strip is rejected: the graph thread is PRESERVED (the downgrade upsert is skipped), the file is HEALED back from the entry (canonical frontmatter; the file's prose body is kept — state lives in the entry JSON, not the body), any path-derived dead dup is deleted, and the heal is logged + counted (`threadHealCount()`). Heal is idempotent — the healed file re-derives as a healthy thread, so the next sync is a normal no-strip upsert. **Invariant the hollow branch rests on:** an OPEN thread ALWAYS carries a `next_step` (the s2a anti-hollow write-discipline — the resume litmus fails on an open thread with none), so anything reaching Branch 2 is a strip or a write-discipline violation; preserve correctly wins both. **Escape hatches** for deliberate change (never hand-strip the frontmatter): retire a thread with `status: done` or delete the file; to PARK a thread without a `next_step`, set `status: pending` or `blocked` (not `open`).

**Code-graph edges (Symbol → Symbol), migration `0007`:**

| Edge | FROM → TO | Semantics |
|---|---|---|
| `CALLS` | Symbol → Symbol | A function/method calls another (in-repo, resolved) |
| `IMPORTS` | Symbol → Symbol | A module imports another module |
| `EXTENDS` | Symbol → Symbol | A class extends another |
| `IMPLEMENTS` | Symbol → Symbol | A class implements an interface |
| `REFERENCES` | Symbol → Symbol | A generic reference between symbols |
| `ABOUT` | Entry → Symbol | A memory is *about* a code symbol (reserved for the future memory↔symbol linker; keeps both tiers in one graph) |

## Frontmatter contract

Every memory `.md` file under `~/.claude/memory/` and `~/.claude/projects/<encoded>/memory/` follows this frontmatter shape. Fields outside this contract are ignored by the sync pipeline. Required fields are starred.

```yaml
---
id: <stable-id>                # optional — derived from path when absent
name: <one-line title>*
description: <one-line summary>*
type: <kind value>*            # see Entry.kind values above
scope: <scope override>        # optional — derived from path when absent

# Authorship + outcome (Phase 1+ writes these; older files default to auto-extracted/empty)
authorship: human | agent | mixed | auto-extracted
outcome: success | failure | unknown
outcome_text: <verbatim outcome>
agent_id: <uuid>               # only when agent-authored
session_id: <claude-session-id>
machine: <machine-id>          # lineage — authoring/first-sync machine. Stamped by ckn-extract; backfill old files with ckn-stamp-lineage.

# Recall + sync controls
pinned: true                   # optional — load-bearing memory: +0.3 recall boost, overrides stale
visibility: local              # optional — when 'local', private-mind never syncs this file off the machine (default: syncs)

# Cross-references (typed edges materialized at sync time)
mentions_files: ["src/foo.ts", "bin/bar.sh"]
mentions_tools: ["Bash", "Edit"]
resolves: ["error:abc-123"]
contradicts: ["memory:old-belief"]
occurred_in: "session:proj/sid"
evolved_from: "memory:older-version"

# Pattern-specific (only for type: pattern)
fingerprint: <8-char hex>
tool: <tool-name>
fail_args: <verbatim>
success_args: <verbatim>
error_text: <verbatim>
---
```

Sync rule: the LLM (when extracting) populates these fields by *reference to transcript content* — file paths come from the transcript, not invented; outcome text is copied byte-for-byte from `tool_result.content`; `mentions_files` lists exactly those files referenced in tool args.

## ID conventions

| Kind | ID format |
|---|---|
| `pattern` | `<encProj>/pattern-<fp>` (file-derived, fingerprint-stable) |
| `concept` (tool) | `concept:<slug>` |
| `file` | `file:<abs-path-with-/-replaced-by-_>` |
| `tool` | `tool:<lowercase-name>` |
| `session` | `session:<encProj>/<sid>` |
| `agent` | `agent:<short-uuid>` |
| `memory`, `decision`, etc | `<encProj>/<filename-no-ext>` (file-derived) |

ID conventions matter because they're how typed edges get materialized — a frontmatter `mentions_files: ["foo.ts"]` translates to a `MENTIONS_FILE` edge to `Entry { id: 'file:foo.ts' }`. If the target Entry doesn't exist yet, sync creates a stub (see "auto-creation" below).

## Auto-created stubs

Some node kinds are auto-created when first referenced rather than waiting for an explicit .md file:

- **File entries** — created when a memory's `mentions_files` lists a path. Stub holds just the path; `kind = 'file'`, `scope = 'file'`.
- **Tool entries** — created when a memory's `mentions_tools` lists a name, or when a pattern's `tool` field references one. Stub: `kind = 'tool'`, `scope = 'tool'`.
- **Session entries** — created when a memory's `occurred_in` references a session that hasn't been recorded yet. Stub: `kind = 'session'`, `scope = 'session:<id>'`.
- **Agent entries** — created when a memory's `agent_id` references an unknown agent. Stub: `kind = 'agent'`, `scope = 'agent'`.

Stubs participate in the graph immediately. Their `.md` file gets written by the next migration pass that consolidates "known referenced entities" — keeps disk authoritative without forcing every reference to come with a file write.

## Outcomes and authorship semantics

The two new fields shape both retrieval ranking and curation:

**Authorship** — the recall hook can prefer human-authored memories when they're equally relevant; the curation UI surfaces human-authored decisions as higher-priority promotion candidates.

| Value | Meaning | Set by |
|---|---|---|
| `human` | Written by the user via `/cortex-snapshot` or hand-edit | `/cortex-snapshot` flow, manual writes |
| `agent` | Written by an autonomous agent (Phase 9 worker) | Orchestrator sets `agent_id` env var; extraction reads it |
| `mixed` | Co-authored or human-curated agent output | Orchestrator's curation step (Phase 10) |
| `auto-extracted` | Produced by `ckn-extract.ts` from a session JSONL | Default for Phase 1 outputs; legacy memories migrate to this |

**Outcome** — the curation gate is `outcome = 'success' AND authorship IN ('human', 'mixed')` for the strictest publication tier. Local recall returns failure memories too because anti-patterns are useful; promotion to cortex-mind defaults to success-only.

| Value | Meaning |
|---|---|
| `success` | The action/decision/workflow produced an explicit success signal in the transcript |
| `failure` | An error or rejection occurred and was not subsequently resolved in the same session |
| `unknown` | No clear outcome signal; usually a reference, topic, or note that doesn't have a pass/fail shape |

`outcome_text` carries the *exact bytes* of the relevant tool result, error message, or success confirmation. The LLM never writes here — only deterministic code that lifts the substring from JSONL by tool_use_id.

## Migration policy

- Migrations only add — never reshape an existing column's meaning, never drop a column without a multi-version deprecation path
- Every migration is idempotent and recorded in `~/.config/ckn/migrations.json`
- ALTER TABLE failures are warnings, not fatals — SQLite's `ALTER TABLE` only supports a narrow set of operations (e.g. `ADD COLUMN`), so a re-applied or unsupported DDL change is tolerated and Cortex degrades gracefully

| Migration | Effect |
|---|---|
| `0001-backfill-pattern-md` | Pattern + concept nodes get .md backing |
| `0002-typed-schema` | Adds typed edge tables, `Pattern` specialization table, new `Entry` columns (authorship, outcome, outcome_text, agent_id, session_id) |
| `0003-session-table` | Adds `Session` specialization table for Claude Code session lifecycle metadata |
| `0004-observation-table` | Adds `Observation` specialization table + `DERIVED_FROM` edge for auto-consolidated beliefs |
| `0005-entry-pinned` | Adds `pinned` BOOL to `Entry` (load-bearing memories, recall boost) |
| `0006-entry-machine` | Adds `machine` STRING to `Entry` (multi-machine lineage) |
| `0007-code-graph-schema` | Adds the `Symbol` node table + code-graph edges (`CALLS`/`IMPORTS`/`EXTENDS`/`IMPLEMENTS`/`REFERENCES`) + `ABOUT` (Entry→Symbol) — the AST/code-graph tier |
| `0008-session-bus` | Adds presence columns to `Session` (`friendly_name`/`cwd`/`machine`/`title`/`last_seen`/`status`/`supersedes`) and creates the `BusMessage` node table for session-to-session communication |

## What this enables

With the schema in place, downstream phases get clean shapes to query:

- **Phase 3 (embeddings)** — embedding column on `Entry`; vector cosine + scope filter
- **Phase 4 (graph-augmented retrieval)** — typed-edge multi-hop queries (`MATCH (e)-[:RESOLVES]->(err) WHERE err.tool = 'Bash'`)
- **Phase 6 (contradictions)** — `CONTRADICTS` edge already wired; detection just populates it
- **Phase 7 (worker)** — `agent_id` + `authorship='agent'` distinguishes worker output
- **Phase 10 (promotion UI)** — query `WHERE outcome = 'success' AND authorship IN ('human', 'mixed')` ranks promotion candidates

The schema is final-shape, not transitional. Future phases populate fields and add edges; they do not reshape what's here.
