# Worker-mode deployment

Cortex on a separate Linux box reachable from your dev workstation over a VLAN. The worker becomes a substrate for autonomous Claude Code sessions sharing one graph.

### One-shot install (recommended)

After cloning the repo and running `npm install`, a single CLI does the rest — generates the systemd user unit, enables linger, starts the service, and optionally bootstraps the shared mind:

```bash
git clone git@github.com:cz-zwtech/cortex.git ~/cortex
cd ~/cortex
npm install
npx tsx bin/ckn-install-worker.ts --remote git@github.com:<your-org>/your-shared-mind.git
```

What it does:
- Validates platform (Linux + systemd; bails on macOS)
- Ensures `bin/cortex-runner.sh` is executable
- Runs `sudo loginctl enable-linger <user>` (needed so user services persist across logout)
- Generates `~/.config/systemd/user/cortex.service` pointing at `bin/cortex-runner.sh`
- `systemctl --user daemon-reload && enable --now cortex.service`
- Configures the cortex-mind remote via `/api/shared/remote` and runs an initial sync

Flags:
- `--remote <url>` — configure cortex-mind remote + initial sync. Skip if you don't want shared-mind on this box.
- `--skip-shared` — install systemd unit only; no shared-mind config
- `--dry-run` — print actions, do nothing

The CLI is idempotent — safe to re-run after pulling latest if the unit template changed.

### Multi-user boxes

The installer is **per-user**: it writes the systemd `--user` unit and registers Cortex's hooks + slash commands into the `~/.claude` of the user the service runs as (first boot). On a box with both a service user (e.g. `claude`) and an interactive user (e.g. `alice`), each user needs Cortex's hooks in their **own** `~/.claude`, and each needs a **git identity** for private-mind push (see Prerequisites). Until a `--user`-aware installer lands, wire the interactive user's hooks separately (boot/register Cortex as that user against the shared graph) and set their `git config` identity. Worker-mode docs otherwise assume a single user.

### Why a wrapper script (cortex-runner.sh)

The systemd unit calls `bin/cortex-runner.sh`, which sources nvm and execs `npm start`. The wrapper resolves the active node version at start time, so `nvm install` doesn't break the unit.

### Manual install (if you don't want the CLI)

If you'd rather walk through the steps:

```bash
# Prereqs (see Prerequisites section)
git clone git@github.com:cz-zwtech/cortex.git ~/cortex
cd ~/cortex
npm install
chmod +x bin/cortex-runner.sh

# Linger (so the unit survives logout)
sudo loginctl enable-linger $(whoami)

# Write ~/.config/systemd/user/cortex.service — see template in
# server/hookRegistrar.ts comments or generated by ckn-install-worker.ts
# in --dry-run mode

# Boot the service
systemctl --user daemon-reload
systemctl --user enable --now cortex.service
systemctl --user status cortex.service

# Bootstrap shared mind (optional, headless — no UI needed)
npx tsx bin/ckn-sync-shared.ts --remote git@github.com:<your-org>/your-shared-mind.git
```

### Connecting from your main machine

| URL | Purpose |
|---|---|
| `http://<worker-lan-ip>:1420` | UI in the browser |
| `http://<worker-lan-ip>:3001/api/graph/stats` | API smoke test |
| `http://<worker-lan-ip>:3001/api/recall` | POST recall queries |

**No auth on top.** Cortex assumes a locked-down VLAN. Don't expose to the public internet without putting auth in front first — every API endpoint is unauthenticated, including writes.

### Per-session provenance for autonomous agents

When running multiple Claude Code sessions on the worker, set these per-session in env so extraction tags memories with the agent identity:

```bash
export CKN_AGENT_ID="agent-uuid-here"
export CKN_LINEAR_TICKET="ENG-1234"
export CKN_TASK_BRANCH="agent/eng-1234-fix-thing"
```

Extraction stamps these into every memory's frontmatter so the graph knows which agent on which task produced what. With `CKN_AGENT_ID` set, `authorship` defaults to `agent` automatically.

### Service management

```bash
systemctl --user status cortex.service    # is it running?
systemctl --user restart cortex.service   # picks up code changes after git pull
systemctl --user stop cortex.service      # halt
journalctl --user -u cortex.service -f    # live logs (alternative to tailing cortex.log)
tail -f ~/cortex/cortex.log               # what the unit's StandardOutput appends to
```

After `git pull`, restart with `systemctl --user restart cortex.service`. Migrations run on boot (idempotent; recorded in `~/.config/ckn/migrations.json`).
